## CRITICAL

1) **DNS rebinding / origin validation bypass (`starts_with`)**
- **Where:** `src/api/mcp_http_server.rs::validate_origin`
- **What:** `origin_str.starts_with("http://localhost")` (and similar) incorrectly allows origins like:
  - `http://localhost.evil.com`
  - `https://127.0.0.1.evil.com`
- **Impact:** Defeats the primary browser-based protection against cross-site access to a localhost wallet service. This is exactly the class of mistake DNS-rebinding defenses try to avoid.
- **Fix:** Parse `Origin` as a URL and compare the **host** exactly (`localhost`, `127.0.0.1`, `[::1]`) and optionally the scheme. Do not use prefix matching. Consider also validating the `Host` header and/or the remote peer IP (loopback only).
- **Missing tests:** Add tests that assert `Origin: http://localhost.evil.com` is rejected (403).

2) **Spending policy bypass for `pay_x402` when `max_amount` is omitted**
- **Where:** `src/api/mcp_router.rs::handle_pay_x402`
- **What:** If `max_amount` is missing, `policy_amount` becomes `"0"` and is used for `check_policy_and_reserve_atomic`.
- **Impact:** The policy check can trivially pass while the actual x402 service price could be non-zero (or much larger). If the CLI path is wired, this can result in **unbounded/unreviewed spend** relative to the configured caps.
- **Fix:** Require a price lookup first (`get_x402_details`), then policy-check the actual quoted price (or require `max_amount` and treat it as a hard cap enforced before payment).
- **Missing tests:** x402 pay without `max_amount` must either (a) fail validation or (b) perform a quote + enforce quote ≤ max.

3) **Potential massive overspend via `trade_tokens` amount semantics (valuation mismatch)**
- **Where:** `src/api/mcp_router.rs::handle_trade_tokens`
- **What:** Policy is checked using the raw `amount` string in the *source asset units*. If your caps are effectively “USD/USDC” (as they appear to be elsewhere), `1.0 ETH` could be treated like `1.0 USDC` for approval.
- **Impact:** If caps are intended to be fiat/USDC-denominated (common for “spending policy”), this is a **direct cap bypass leading to loss of funds**.
- **Fix:** Define policy units explicitly. If caps are in USDC, convert trade amount to USDC-equivalent (quote) before policy check. If caps are per-asset units, enforce per-asset caps and store the asset in policy accounting.
- **Missing tests:** A trade of `1 ETH` with a small cap should be denied if caps are USDC-based.

4) **CLI timeout does not terminate the spawned process (unknown transaction outcome)**
- **Where:** `src/cli/executor.rs::RealCliExecutor::run`
- **What:** On `tokio::time::timeout`, returns `CliError::Timeout` but does **not** kill the child process.
- **Impact:** The CLI process may continue and broadcast a transaction after the server reports failure/timeout, prompting retries and **double-spend risk** plus orphaned processes.
- **Fix:** Keep `child` mutable and on timeout call `child.kill().await` (and ideally `child.wait().await`), then return timeout. Also consider idempotency keys for sends.
- **Missing tests:** Simulate a long-running command and assert the process is killed on timeout.

## HIGH

1) **Token hashing contradicts config (`argon2id`) and uses unsalted SHA-256**
- **Where:** `src/api/mcp_http_server.rs::validate_bearer_token`, `src/api/mcp_server.rs::validate_token_with_cli`, `src/config.rs` (`token_hash_algorithm`)
- **What:** Config says `argon2id`, but implementation uses plain SHA-256 hex without salt.
- **Impact:** If the DB leaks, tokens are much easier to brute-force/offline attack than intended. (Even if tokens are high entropy, this is still a needless downgrade.)
- **Fix:** Use Argon2id (or at least a salted KDF) and store per-token salts; or store only a randomly-generated opaque token with a strong hash + salt.

2) **No rate limiting / brute-force controls despite config knobs**
- **Where:** Server logic: no enforcement for `rate_limit_requests_per_minute`, `token_cache_ttl_seconds`, invitation guessing, etc.
- **Impact:** Enables high-volume token guessing (online) and invitation code guessing; also enables resource DoS (sessions, request bodies).
- **Fix:** Implement per-IP (loopback still matters) and per-token rate limits; consider exponential backoff on auth failures.

3) **Sessions never expire and are unbounded (memory DoS)**
- **Where:** `src/api/mcp_http_server.rs` (`DashMap<String, McpSession>`), `handle_initialize`
- **What:** Every `initialize` inserts a session; there is no TTL, max sessions, or cleanup.
- **Impact:** Any local process (or any browser origin that passes the flawed origin check) can exhaust memory by spamming `initialize`.
- **Fix:** Add TTL + periodic cleanup, cap total sessions, and/or require auth before allocating server resources.

4) **No request/body size limits**
- **Where:** `handle_post(State, headers, body: String)`
- **What:** Accepts an arbitrarily large body into memory.
- **Impact:** Simple DoS via large POST to localhost.
- **Fix:** Use Axum `DefaultBodyLimit` / `RequestBodyLimitLayer`, reject bodies above a safe threshold.

5) **Async handler performs heavy sync work (DB + CLI orchestration)**
- **Where:** `src/api/mcp_http_server.rs::handle_tools_call` calls `router.handle_tool_call(...)` synchronously; router does DB calls and blocks waiting for CLI thread join.
- **Impact:** Can block Tokio worker threads under load, degrading responsiveness and amplifying DoS.
- **Fix:** Make tool handling async end-to-end or wrap blocking tool execution in `tokio::task::spawn_blocking` and return results asynchronously.

6) **Protocol version inconsistency between HTTP and stdio MCP**
- **Where:** HTTP: `PROTOCOL_VERSION = "2025-11-25"`; stdio initialize returns `"2024-11-05"` in `src/api/mcp_server.rs::handle_request`
- **Impact:** Clients may refuse to connect or behave unpredictably.
- **Fix:** Use one protocol constant across transports; validate the client’s requested version and error cleanly on mismatch.

7) **Stdio JSON-RPC notifications handled incorrectly**
- **Where:** `src/api/mcp_server.rs::run` + `handle_request`
- **What:** Stdio server always emits a response; JSON-RPC notifications (no `id`) should produce **no response**.
- **Impact:** Breaks clients that rely on notification semantics.
- **Fix:** If `request.id.is_none()`, do not write a response.

8) **Auth/agent status enforcement is unclear/incomplete at the auth boundary**
- **Where:** `validate_bearer_token(...)` returns agent id only; `McpServer::validate_token_with_cli` does not check `AgentStatus`.
- **Impact:** If `check_policy_and_reserve_atomic` does not enforce agent status centrally, suspended/revoked agents could still transact.
- **Fix:** Enforce `AgentStatus::Active` during token validation (HTTP + stdio), not only in one constructor path.
- **Missing tests:** A suspended agent’s valid token must be rejected before tool execution.

9) **Public tools listing ignores intended “unauthenticated tool subset”**
- **Where:** HTTP `handle_tools_list` always returns all tools; router has `list_tools(authenticated: bool)` but it’s unused.
- **Impact:** Not a direct vuln, but it undermines the designed auth UX and can mislead clients into attempting calls that will fail.
- **Fix:** For unauthenticated sessions, return only `register_agent` until a valid token is presented.

## MEDIUM

1) **Origin header optional = inconsistent browser security posture**
- **Where:** `validate_origin` allows missing `Origin`.
- **Impact:** Some cross-site request types and non-browser contexts may omit `Origin`; relying on “Origin if present” is fragile for a financial endpoint.
- **Fix:** Consider requiring additional headers (`Sec-Fetch-Site`, `Sec-Fetch-Mode`) for browser contexts, or require a locally-issued CSRF-style nonce for session creation.

2) **HTTP error responses are often plain text, not JSON-RPC shaped**
- **Where:** missing session id (400), session not found (404), invalid accept (400), origin forbidden (403)
- **Impact:** Clients expecting JSON-RPC error objects may not handle these well. MCP Streamable HTTP may allow transport-level HTTP errors, but you should be consistent with the spec.
- **Fix:** Confirm MCP 2025-11-25 requirements: if JSON-RPC body is required for POST failures, return JSON-RPC errors (e.g., -32600) with `id` echoed when possible.

3) **`initialize` doesn’t validate request `jsonrpc` field or requested protocol**
- **Where:** HTTP and stdio
- **Impact:** Looser parsing increases compatibility issues and makes it harder to reason about correctness.
- **Fix:** Validate `jsonrpc == "2.0"`, and validate/echo negotiated protocol version.

4) **Invitation code usage enforcement likely non-atomic**
- **Where:** `handle_register_agent` checks `use_count >= max_uses` but no visible increment/update in this code
- **Impact:** Concurrent registrations can exceed `max_uses`. Also enables reuse if `use_count` is never incremented.
- **Fix:** Enforce max-uses with a single atomic DB update (e.g., `UPDATE ... WHERE use_count < max_uses`).
- **Missing tests:** Concurrent register calls should not exceed `max_uses`.

5) **`get_transactions` limit is unbounded**
- **Where:** `handle_get_transactions` reads `limit` from user, default 10, no max cap
- **Impact:** Local DoS via huge limit.
- **Fix:** Clamp limit (e.g., max 100).

6) **Session `agent_id` stored but unused**
- **Where:** `McpSession { agent_id }` updated in `handle_register_agent_call`, but not enforced later
- **Impact:** Confusing and creates a false sense of binding between session and agent.
- **Fix:** Either bind session to agent after auth (and require token consistency), or remove `agent_id` from session.

## LOW

1) **Overly strict Accept header requirement**
- **Where:** `handle_post` rejects if `Accept` doesn’t contain `application/json`
- **Impact:** Some clients send `Accept: */*` and would fail unnecessarily.
- **Fix:** Prefer validating request `Content-Type` and accept broader `Accept` values unless MCP spec mandates it.

2) **Tool results always encoded as `"type":"text"` with JSON string**
- **Where:** `handle_tools_call` / `handle_request` wrap tool JSON as `content.to_string()`
- **Impact:** Works, but reduces structured interoperability (clients must parse a JSON string).
- **Fix:** If MCP supports structured content types, return JSON content directly or add a `type: "json"` variant.

3) **Hardcoded server name/version scattered**
- **Where:** constants in HTTP; literals in stdio
- **Fix:** centralize constants.
