## CRITICAL

### 1) Spending-policy ‚Äúreserve‚Äù is not reconciled on execution failure / non-execution
**Where:** `src/api/mcp_router.rs` (`handle_send_payment`, `handle_trade_tokens`, `handle_pay_x402`)  
These handlers call `queries::check_policy_and_reserve_atomic(...)` and then:
- **If CLI is absent:** mark the transaction `Pending` and return success.
- **If CLI is present but execution fails:** mark `Failed` and return success.

There is **no visible rollback/release** of the ‚Äúreserved‚Äù spending when:
- the CLI execution fails, or
- the transfer is never executed (the ‚ÄúPending with no CLI‚Äù path), or
- execution succeeds but parsing yields no `tx_hash`.

If `check_policy_and_reserve_atomic` actually increments/locks budget (as the name strongly implies), this can **permanently consume caps** and block future legitimate transactions (or create accounting drift). For financial controls, this is a top-tier correctness/safety issue.

**Fix:** Make ‚Äúreserve‚Äù tie to a transaction row and have deterministic reconciliation:
- On CLI success: finalize reservation ‚Üí ‚Äúspent‚Äù.
- On CLI failure: release reservation (or mark ‚Äúfailed‚Äù and exclude from cap).
- If CLI is not configured: **do not reserve** (or return an explicit error), unless there is a background executor that will later finalize.

### 2) Auth bypass for suspended/pending/revoked agents (status not enforced on token auth)
**Where:**
- HTTP: `src/api/mcp_http_server.rs` ‚Üí `validate_bearer_token()`
- stdio: `src/api/mcp_server.rs` ‚Üí `validate_token_with_cli()`

`validate_bearer_token()` returns an `agent_id` purely by token-hash lookup and does **not** check `AgentStatus`.  
`McpServer::validate_token_with_cli()` similarly binds a server to the agent without checking status (contrast with `new_with_agent_id_and_cli()` which *does* enforce Active).

Impact: a **suspended** (or otherwise non-active) agent token can still be used to access tools and potentially spend.

**Fix:** After lookup by token hash, fetch/check the agent record and require `status == Active` (and not revoked), returning a clear auth error otherwise.

### 3) MCP HTTP server in the shipped app is started without the CLI executor (financial ops silently don‚Äôt execute)
**Where:** `src/lib.rs` (MCP spawn), `src/api/mcp_http_server.rs`, `src/api/mcp_router.rs`

In `lib.rs`, the MCP HTTP server is spawned with:
```rust
let mcp_state = McpHttpState::new(mcp_db); // cli = None
```
So all MCP calls go through a router with `cli: None`, triggering the ‚ÄúPending (not executed)‚Äù path for transfers/trades/x402 payments. This is effectively **a silent operational failure**: the system records ‚Äúpending‚Äù transactions but does not actually broadcast anything, and there‚Äôs no background worker shown that later executes pending actions.

**Fix:** Start MCP HTTP with `new_with_cli(app_state.db.clone(), app_state.cli.clone())` (or equivalent), and/or fail closed for spending tools when CLI is not configured.

## HIGH

### 4) Race-to-panic DoS in session rate limiting (`unwrap()` after non-atomic check)
**Where:** `src/api/mcp_http_server.rs` in `handle_post` rate-limit section
You first `get()` the session, then later do:
```rust
let mut session = state.sessions.get_mut(&session_id).unwrap();
```
Between those points, another request (e.g. `DELETE /mcp`) can remove the session, causing a panic. Depending on panic strategy/build, this can be a **reliable DoS**.

**Fix:** Use a single `get_mut()` and handle `None` safely; never `unwrap()` on attacker-controlled concurrency paths.

### 5) Invitation codes are not consumed (max_uses enforcement likely ineffective)
**Where:** `src/api/mcp_router.rs` ‚Üí `handle_register_agent`
You check:
```rust
if code.use_count >= code.max_uses { ... }
```
‚Ä¶but do not increment `use_count` (nor mark the code as used) anywhere in this flow. Unless the DB layer does it implicitly (not shown), this enables **unlimited registrations** with the same invitation code.

**Fix:** Atomically increment `use_count` (or mark as redeemed) in the same transaction that creates the agent.

### 6) `register_agent` does not return a token, but docs/instructions claim it does; agents cannot authenticate via MCP
**Where:**
- `src/core/auto_discovery.rs` instructions: ‚ÄúSave the returned token‚Ä¶‚Äù
- `src/api/mcp_router.rs` ‚Üí `handle_register_agent`: creates agent with `api_token_hash: None` and returns only `agent_id`

This is a major functional/security flow break: new agents can‚Äôt obtain a bearer token from MCP, yet all other tools require it.

**Fix:** Decide the intended lifecycle:
- If registration should return a token: generate a high-entropy token, store a strong hash, return token once.
- If registration is approval-gated: update instructions and expose a separate ‚Äúget token after approval‚Äù flow (or UI-only).

### 7) Token security is weak/inconsistent: SHA-256 unsalted hashing hardcoded despite config indicating Argon2id
**Where:**
- `src/api/mcp_http_server.rs` ‚Üí `validate_bearer_token()` uses SHA-256
- `src/api/mcp_server.rs` ‚Üí `validate_token_with_cli()` uses SHA-256
- `src/config.rs` default: `token_hash_algorithm: "argon2id"`

Problems:
- Hardcoded SHA-256 is **not** a password-hash; if tokens are ever low entropy, offline guessing is feasible.
- The presence of `token_hash_algorithm` suggests the rest of the system may expect Argon2id hashes, making MCP auth potentially incompatible.

**Fix:** Centralize token hashing/verification behind one module honoring `token_hash_algorithm`, and use a proper PHC (Argon2id) with per-token salt unless you can guarantee 256-bit random tokens.

### 8) Financial error handling is lossy: CLI failures don‚Äôt persist error details; ‚Äúconfirmed‚Äù may be reported without a tx hash
**Where:** `src/api/mcp_router.rs` in send/trade/x402 ‚ÄúAutoApproved‚Äù branches
- On CLI error you set `(None, TxStatus::Failed)` but store `error_message: None` and return success with `"status": "failed"`.
- On CLI success you set `TxStatus::Confirmed` even if `tx_hash` is absent/unparseable.

This harms auditability and can confuse downstream automation.

**Fix:** Persist `error_message` on failure; require a tx hash (or other strong success indicator) before marking ‚Äúconfirmed‚Äù.

### 9) Rate limiting is per-session only; easy amplification by opening many sessions
**Where:** `src/api/mcp_http_server.rs`
With `MAX_SESSIONS=100` and `RATE_LIMIT_MAX=60`, a single local attacker can generate ~6000 req/min by rotating sessions. For a process-spawning backend, that‚Äôs substantial load and could become a practical DoS.

**Fix:** Add global + per-token/per-agent rate limits (and/or require auth for high-cost tools, plus concurrency limits on CLI execution).

## MEDIUM

### 10) MCP 2025-11-25 Streamable HTTP compliance gaps (SSE not implemented; error semantics not consistently JSON-RPC)
**Where:** `src/api/mcp_http_server.rs`
- `GET /mcp` returns `405` (‚ÄúSSE not implemented‚Äù) while file header claims Streamable HTTP transport.
- Several error paths return plain text with non-200 HTTP codes (e.g., missing session id ‚Üí `400` text) instead of JSON-RPC error objects (many clients expect JSON-RPC envelope even on transport errors).
- `initialize` does not validate the requested `protocolVersion`.

**Fix:** Implement SSE (or downgrade claim), validate protocol version, and standardize error responses per MCP Streamable HTTP expectations.

### 11) Origin validation is incomplete/buggy for IPv6 and relies solely on `Origin`
**Where:** `src/api/mcp_http_server.rs` ‚Üí `validate_origin()`
- `url.host_str()` for `http://[::1]:...` returns `"::1"`, but code allows `"[::1]"`, so legitimate IPv6 localhost origins may be rejected.
- No `Host` header validation (likely fine given bind to 127.0.0.1, but still a gap if binding ever changes).

**Fix:** Allow `"::1"`, and consider validating `Host` as defense-in-depth.

**Fix:** Treat `register_agent` like other tools: run via `spawn_blocking` (or use async DB).
