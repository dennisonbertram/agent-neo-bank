## 1. Tauri's Built-in Updater Plugin

Tauri v2 uses `tauri-plugin-updater` (replacing the built-in updater from v1). This is a separate plugin that must be explicitly installed and configured.

### Installation

**Rust (Cargo.toml)**:
```bash
cargo add tauri-plugin-updater --target 'cfg(any(target_os = "macos", windows, target_os = "linux"))'
```

**JavaScript/TypeScript**:
```bash
npm install @tauri-apps/plugin-updater
```

**Minimum Rust version**: 1.77.2+

### Plugin Registration (Rust)

In `src-tauri/src/lib.rs`:

```rust
use tauri_plugin_updater::UpdaterExt;

pub fn run() {
    tauri::Builder::default()
        // Only register on desktop platforms
        .plugin(tauri_plugin_updater::Builder::new().build())
        .setup(|app| {
            let handle = app.handle().clone();
            tauri::async_runtime::spawn(async move {
                if let Err(e) = update(handle).await {
                    eprintln!("Update check failed: {e}");
                }
            });
            Ok(())
        })
        .run(tauri::generate_context!())
        .unwrap();
}
```

### Configuration (tauri.conf.json)

```json
{
  "bundle": {
    "createUpdaterArtifacts": true
  },
  "plugins": {
    "updater": {
      "pubkey": "CONTENT_OF_YOUR_PUBLIC_KEY",
      "endpoints": [
        "https://github.com/user/repo/releases/latest/download/latest.json"
      ]
    }
  }
}
```

Key fields:
- **`createUpdaterArtifacts`**: Must be `true` for v2. Set to `"v1Compatible"` only when migrating from Tauri v1.
- **`pubkey`**: The public key generated by the Tauri signer CLI. Used to verify update signatures.
- **`endpoints`**: Array of URLs. Supports dynamic variables: `{{current_version}}`, `{{target}}`, `{{arch}}`.

### Permissions (capabilities/default.json)

```json
{
  "permissions": [
    "updater:default"
  ]
}
```

The `updater:default` permission grants:
- `allow-check` -- check for updates
- `allow-download` -- download update files
- `allow-install` -- install updates
- `allow-download-and-install` -- combined operation

For more granular control, individual permissions can be specified (e.g., `updater:deny-install` to block auto-install).

### Frontend API (TypeScript)

```typescript
import { check } from '@tauri-apps/plugin-updater';
import { relaunch } from '@tauri-apps/plugin-process';

const update = await check();
if (update) {
  console.log(`Update ${update.version} available (notes: ${update.body})`);

  let downloaded = 0;
  let contentLength = 0;

  await update.downloadAndInstall((event) => {
    switch (event.event) {
      case 'Started':
        contentLength = event.data.contentLength;
        break;
      case 'Progress':
        downloaded += event.data.chunkLength;
        console.log(`Progress: ${Math.round((downloaded / contentLength) * 100)}%`);
        break;
      case 'Finished':
        console.log('Download complete');
        break;
    }
  });

  await relaunch();
}
```

The `check()` function also accepts optional configuration:

```typescript
const update = await check({
  proxy: '<proxy url>',
  timeout: 30000,
  headers: { Authorization: 'Bearer <token>' }
});
```

### Backend API (Rust)

```rust
use tauri_plugin_updater::UpdaterExt;

async fn update(app: tauri::AppHandle) -> tauri_plugin_updater::Result<()> {
    if let Some(update) = app.updater()?.check().await? {
        let mut downloaded = 0;

        update
            .download_and_install(
                |chunk_length, content_length| {
                    downloaded += chunk_length;
                    println!("downloaded {downloaded} from {content_length:?}");
                },
                || {
                    println!("download finished");
                },
            )
            .await?;

        app.restart();
    }
    Ok(())
}
```

You can also separate download and install into two steps:

```rust
// Step 1: Download only
update.download(/* progress callback */).await?;

// Step 2: Install (can happen later, e.g., on user confirmation)
update.install()?;
```

## 2. Update Distribution Methods

### 2a. GitHub Releases (Recommended for Open Source)

The simplest approach. The `tauri-action` GitHub Action generates a `latest.json` file automatically and uploads it alongside binaries to a GitHub Release.

**Endpoint configuration**:
```json
"endpoints": [
  "https://github.com/user/repo/releases/latest/download/latest.json"
]
```

**`latest.json` format (static, multi-platform)**:
```json
{
  "version": "1.2.0",
  "notes": "Bug fixes and performance improvements",
  "pub_date": "2026-02-28T12:00:00Z",
  "platforms": {
    "darwin-x86_64": {
      "signature": "SIGNATURE_CONTENT_FROM_SIG_FILE",
      "url": "https://github.com/user/repo/releases/download/v1.2.0/MyApp.app.tar.gz"
    },
    "darwin-aarch64": {
      "signature": "SIGNATURE_CONTENT",
      "url": "https://github.com/user/repo/releases/download/v1.2.0/MyApp-aarch64.app.tar.gz"
    },
    "linux-x86_64": {
      "signature": "SIGNATURE_CONTENT",
      "url": "https://github.com/user/repo/releases/download/v1.2.0/MyApp.AppImage"
    },
    "windows-x86_64": {
      "signature": "SIGNATURE_CONTENT",
      "url": "https://github.com/user/repo/releases/download/v1.2.0/MyApp-setup.exe"
    }
  }
}
```

Platform key format: `{os}-{arch}` where:
- OS: `darwin`, `linux`, `windows`
- Arch: `x86_64`, `aarch64`, `i686`, `armv7`

### 2b. Custom Update Server

For dynamic updates (A/B testing, phased rollouts, analytics).

**Dynamic endpoint format**:
```json
"endpoints": [
  "https://releases.myapp.com/{{target}}/{{arch}}/{{current_version}}"
]
```

Server should respond:
- **200 OK** with update JSON when an update is available
- **204 No Content** when no update exists

**Dynamic response format**:
```json
{
  "version": "1.2.0",
  "pub_date": "2026-02-28T12:00:00Z",
  "url": "https://cdn.myapp.com/releases/v1.2.0/MyApp-setup.exe",
  "signature": "SIGNATURE_CONTENT",
  "notes": "Release notes here"
}
```

### 2c. CrabNebula Cloud (Managed CDN)

A managed update distribution service built specifically for Tauri apps.

**Endpoint**:
```json
"endpoints": [
  "https://cdn.crabnebula.app/update/org/app/{{target}}-{{arch}}/{{current_version}}"
]
```

Advantages:
- No self-hosted infrastructure
- Built-in CDN and analytics
- Handles signature verification
- Works with existing CI/CD

### 2d. S3/CDN (Self-Hosted)

Host `latest.json` and binaries on S3, CloudFront, Cloudflare R2, etc.

```json
"endpoints": [
  "https://updates.myapp.com/latest.json"
]
```

Upload the `latest.json` and all platform binaries + `.sig` files to the same bucket/CDN.

### Endpoint Fallback

If multiple endpoints are specified, Tauri falls back to the next one if a server does not respond within the configured timeout. This provides redundancy:

```json
"endpoints": [
  "https://primary-cdn.myapp.com/latest.json",
  "https://github.com/user/repo/releases/latest/download/latest.json"
]
```

## 3. Signing Updates

Tauri **requires** cryptographic signing for all updates. This is not optional -- unsigned updates will be rejected.

### Key Generation

```bash
npx tauri signer generate -w ~/.tauri/myapp.key
```

This produces:
1. **Private key** (`~/.tauri/myapp.key`) -- Used to sign builds. NEVER share this. If lost, existing users cannot receive updates.
2. **Public key** (`~/.tauri/myapp.key.pub`) -- Placed in `tauri.conf.json`. Safe to share.

You will be prompted for a password during generation.

### Build-Time Signing

Before running `tauri build`, set these environment variables:

```bash
export TAURI_SIGNING_PRIVATE_KEY="path/to/key OR key content"
export TAURI_SIGNING_PRIVATE_KEY_PASSWORD="your-password"
```

When `createUpdaterArtifacts` is `true`, the build produces `.sig` files alongside each installer:
- `MyApp-setup.exe` + `MyApp-setup.exe.sig`
- `MyApp.app.tar.gz` + `MyApp.app.tar.gz.sig`
- `MyApp.AppImage` + `MyApp.AppImage.sig`

### Signature Verification Flow

1. App downloads `latest.json` from endpoint
2. Reads the `signature` field for its platform
3. Downloads the update binary from the `url`
4. Verifies the binary against the signature using the embedded public key
5. Only installs if verification passes

### Runtime Public Key Override

For advanced key rotation scenarios:

```rust
use tauri_plugin_updater::UpdaterExt;

let update = app
    .updater_builder()
    .pubkey("<new-public-key>")
    .build()?
    .check()
    .await?;
```

### macOS Code Signing and Notarization

macOS requires additional Apple-specific signing beyond Tauri's update signing:

1. **Developer ID Certificate** -- Required for distribution outside the App Store
2. **Notarization** -- Apple scans the binary and issues a notarization ticket

Required CI/CD secrets for macOS:
- `APPLE_CERTIFICATE` (base64 encoded .p12)
- `APPLE_CERTIFICATE_PASSWORD`
- `APPLE_SIGNING_IDENTITY` (e.g., "Developer ID Application: Your Name (TEAMID)")
- `APPLE_ID` (Apple developer email)
- `APPLE_PASSWORD` (app-specific password, NOT your Apple ID password)
- `APPLE_TEAM_ID`

### Windows Code Signing

Windows signing typically uses:
- EV code signing certificates (hardware token)
- Azure Key Vault with the `relic` signing tool

Required CI/CD secrets for Azure Key Vault:
- `AZURE_CLIENT_ID`
- `AZURE_TENANT_ID`
- `AZURE_CLIENT_SECRET`

## 4. Update Checking Strategies

### 4a. On Launch (Most Common)

Check for updates when the app starts, in the Rust `setup` hook:

```rust
.setup(|app| {
    let handle = app.handle().clone();
    tauri::async_runtime::spawn(async move {
        // Small delay to let the UI load first
        tokio::time::sleep(std::time::Duration::from_secs(3)).await;
        if let Err(e) = check_for_update(handle).await {
            eprintln!("Update check failed: {e}");
        }
    });
    Ok(())
})
```

### 4b. Periodic Background Checks

Use a timer to check at regular intervals:

```typescript
import { check } from '@tauri-apps/plugin-updater';

// Check every 4 hours
const UPDATE_INTERVAL = 4 * 60 * 60 * 1000;

function startUpdateChecker() {
  setInterval(async () => {
    try {
      const update = await check();
      if (update) {
        showUpdateNotification(update);
      }
    } catch (e) {
      console.error('Update check failed:', e);
    }
  }, UPDATE_INTERVAL);
}
```

### 4c. User-Initiated

Provide a "Check for Updates" button in Settings:

```typescript
async function manualUpdateCheck() {
  try {
    const update = await check();
    if (update) {
      showUpdateDialog(update);
    } else {
      showNotification('You are on the latest version.');
    }
  } catch (e) {
    showErrorNotification('Failed to check for updates.');
  }
}
```

### 4d. Tauri Command Pattern (Recommended)

Expose update checking as a Tauri command for full Rust-side control:

```rust
use std::sync::Mutex;
use tauri::State;
use tauri_plugin_updater::{Update, UpdaterExt};

struct PendingUpdate(Mutex<Option<Update>>);

#[derive(serde::Serialize)]
struct UpdateMetadata {
    version: String,
    current_version: String,
    body: Option<String>,
    date: Option<String>,
}

#[tauri::command]
async fn check_for_update(
    app: tauri::AppHandle,
    pending: State<'_, PendingUpdate>,
) -> Result<Option<UpdateMetadata>, String> {
    let update = app.updater()
        .map_err(|e| e.to_string())?
        .check()
        .await
        .map_err(|e| e.to_string())?;

    let metadata = update.as_ref().map(|u| UpdateMetadata {
        version: u.version.clone(),
        current_version: u.current_version.clone(),
        body: Some(u.body.clone()),
        date: u.date.map(|d| d.to_string()),
    });

    *pending.0.lock().unwrap() = update;
    Ok(metadata)
}

#[tauri::command]
async fn install_update(
    app: tauri::AppHandle,
    pending: State<'_, PendingUpdate>,
) -> Result<(), String> {
    let update = pending.0.lock().unwrap().take()
        .ok_or("No pending update")?;

    update.download_and_install(
        |chunk, total| { /* progress callback */ },
        || { /* finished callback */ },
    ).await.map_err(|e| e.to_string())?;

    app.restart();
    Ok(())
}
```

## 5. UX Best Practices

### Update Dialog Patterns

**Pattern 1: Non-intrusive notification banner**
```typescript
// Show a dismissible banner at the top of the app
function showUpdateBanner(update: Update) {
  return (
    <div className="update-banner">
      <span>Version {update.version} is available.</span>
      <button onClick={() => installUpdate(update)}>Update Now</button>
      <button onClick={dismiss}>Later</button>
    </div>
  );
}
```

**Pattern 2: Modal dialog for critical updates**
```typescript
// For security patches or breaking changes
function showCriticalUpdateDialog(update: Update) {
  return (
    <Dialog open={true} onClose={() => {}}>
      <h2>Critical Update Required</h2>
      <p>Version {update.version} contains important security fixes.</p>
      <Button onClick={() => installUpdate(update)}>Update Now</Button>
    </Dialog>
  );
}
```

### Progress Indicator

```typescript
function UpdateProgress({ update }: { update: Update }) {
  const [progress, setProgress] = useState(0);
  const [status, setStatus] = useState<'idle' | 'downloading' | 'installing'>('idle');

  async function handleInstall() {
    setStatus('downloading');
    let downloaded = 0;

    await update.downloadAndInstall((event) => {
      if (event.event === 'Started') {
        // contentLength may be 0 if server doesn't send Content-Length
      }
      if (event.event === 'Progress') {
        downloaded += event.data.chunkLength;
        setProgress(downloaded);
      }
      if (event.event === 'Finished') {
        setStatus('installing');
      }
    });

    await relaunch();
  }

  return (
    <div>
      <p>Downloading update...</p>
      <ProgressBar value={progress} />
      <p>{status === 'installing' ? 'Installing...' : 'Downloading...'}</p>
    </div>
  );
}
```

### Recommended UX Flow

1. **Silent check on startup** (no UI unless update found)
2. **Non-blocking notification** for optional updates
3. **Modal dialog** for critical/security updates
4. **Progress indicator** during download
5. **Auto-restart** after install (with brief countdown or user confirmation)
6. **"Check for Updates" in Settings** for user-initiated checks -- only show "up to date" message here
7. **Don't pester** -- if user dismisses, don't ask again until next version

## 6. Platform-Specific Considerations

### macOS

| Aspect | Detail |
|--------|--------|
| **Artifact** | `.app.tar.gz` + `.sig` |
| **Restart** | Manual restart required (app does NOT auto-exit) |
| **Code signing** | Developer ID certificate required for Gatekeeper |
| **Notarization** | Required for macOS 10.15+, adds 2-5 minutes to CI |
| **Universal binary** | Build separate `x86_64` and `aarch64` or use custom target `macos-universal` |

```rust
// Custom target for universal binary
let update = app
    .updater_builder()
    .target("macos-universal")
    .build()?
    .check()
    .await?;
```

### Windows

| Aspect | Detail |
|--------|--------|
| **Artifacts** | `.exe` (NSIS) or `.msi` + `.sig` |
| **Restart** | **Auto-exits before install** (limitation of Windows installers) |
| **Install modes** | `passive` (default), `basicUi`, `quiet` |
| **Code signing** | EV certificate recommended (avoids SmartScreen warnings) |

```json
{
  "plugins": {
    "updater": {
      "windows": {
        "installMode": "passive"
      }
    }
  }
}
```

Install modes:
- **`passive`** (default): Shows a small progress bar, no user interaction needed
- **`basicUi`**: Shows the full NSIS installer UI
- **`quiet`**: Completely silent, no UI at all

**Windows before-exit hook**:
```rust
app.updater_builder()
    .on_before_exit(|| {
        // Save state, close connections, etc.
        println!("App exiting for Windows update installation");
    })
    .build()?
```

### Linux

| Aspect | Detail |
|--------|--------|
| **Artifact** | `.AppImage` + `.sig` |
| **Restart** | Manual restart required |
| **Code signing** | Tauri signature only (no OS-level signing) |
| **AppImage** | Self-contained, replaces in place |

### Mobile (Android/iOS)

The updater plugin is **not available** on mobile platforms. Mobile apps must use their respective app store update mechanisms.

## 7. CI/CD Integration

### GitHub Actions Workflow (Complete Example)

```yaml
name: Release

on:
  push:
    tags:
      - 'v*'

concurrency:
  group: ${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: true

jobs:
  build:
    strategy:
      fail-fast: false
      matrix:
        include:
          - platform: macos-latest
            target: aarch64-apple-darwin
            args: --target aarch64-apple-darwin
          - platform: macos-latest
            target: x86_64-apple-darwin
            args: --target x86_64-apple-darwin
          - platform: ubuntu-22.04
            target: x86_64-unknown-linux-gnu
            args: ''
          - platform: windows-latest
            target: x86_64-pc-windows-msvc
            args: ''

    runs-on: ${{ matrix.platform }}

    steps:
      - uses: actions/checkout@v4

      - name: Install Linux dependencies
        if: matrix.platform == 'ubuntu-22.04'
        run: |
          sudo apt-get update
          sudo apt-get install -y \
            libwebkit2gtk-4.1-dev \
            libappindicator3-dev \
            librsvg2-dev \
            patchelf

      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: 'npm'

      - name: Install Rust toolchain
        uses: dtolnay/rust-toolchain@stable
        with:
          targets: ${{ matrix.target }}

      - uses: swatinem/rust-cache@v2
        with:
          workspaces: src-tauri

      - name: Install frontend dependencies
        run: npm ci

      # macOS code signing
      - name: Import Apple certificate
        if: matrix.platform == 'macos-latest'
        env:
          APPLE_CERTIFICATE: ${{ secrets.APPLE_CERTIFICATE }}
          APPLE_CERTIFICATE_PASSWORD: ${{ secrets.APPLE_CERTIFICATE_PASSWORD }}
        run: |
          CERTIFICATE_PATH=$RUNNER_TEMP/certificate.p12
          KEYCHAIN_PATH=$RUNNER_TEMP/app-signing.keychain-db
          KEYCHAIN_PASSWORD=$(openssl rand -hex 32)

          echo -n "$APPLE_CERTIFICATE" | base64 --decode -o $CERTIFICATE_PATH
          security create-keychain -p "$KEYCHAIN_PASSWORD" "$KEYCHAIN_PATH"
          security default-keychain -s "$KEYCHAIN_PATH"
          security unlock-keychain -p "$KEYCHAIN_PASSWORD" "$KEYCHAIN_PATH"
          security import "$CERTIFICATE_PATH" -P "$APPLE_CERTIFICATE_PASSWORD" \
            -A -t cert -f pkcs12 -k "$KEYCHAIN_PATH"
          security set-key-partition-list -S apple-tool:,apple: \
            -k "$KEYCHAIN_PASSWORD" "$KEYCHAIN_PATH"
          security list-keychain -d user -s "$KEYCHAIN_PATH"

      - name: Build and release
        uses: tauri-apps/tauri-action@v0
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
          # Update signing
          TAURI_SIGNING_PRIVATE_KEY: ${{ secrets.TAURI_SIGNING_PRIVATE_KEY }}
          TAURI_SIGNING_PRIVATE_KEY_PASSWORD: ${{ secrets.TAURI_SIGNING_PRIVATE_KEY_PASSWORD }}
          # macOS notarization
          APPLE_SIGNING_IDENTITY: ${{ secrets.APPLE_SIGNING_IDENTITY }}
          APPLE_ID: ${{ secrets.APPLE_ID }}
          APPLE_PASSWORD: ${{ secrets.APPLE_PASSWORD }}
          APPLE_TEAM_ID: ${{ secrets.APPLE_TEAM_ID }}
        with:
          tagName: v__VERSION__
          releaseName: 'v__VERSION__'
          releaseBody: 'See the assets to download and install.'
          releaseDraft: true
          prerelease: false
          args: ${{ matrix.args }}
```

### Required GitHub Secrets

| Secret | Purpose |
|--------|---------|
| `TAURI_SIGNING_PRIVATE_KEY` | Tauri update signature private key content |
| `TAURI_SIGNING_PRIVATE_KEY_PASSWORD` | Password for the signing key |
| `APPLE_CERTIFICATE` | Base64-encoded .p12 certificate |
| `APPLE_CERTIFICATE_PASSWORD` | Certificate password |
| `APPLE_SIGNING_IDENTITY` | e.g., "Developer ID Application: Name (TEAMID)" |
| `APPLE_ID` | Apple developer email |
| `APPLE_PASSWORD` | App-specific password (NOT Apple ID password) |
| `APPLE_TEAM_ID` | Apple Developer Team ID |

### Repository Settings

- **Actions > General > Workflow permissions**: Must be set to "Read and write permissions" (default read-only causes "Resource not accessible by integration" errors)

### What `tauri-action` Produces

For each platform, the action generates:
- The installer binary (`.exe`, `.msi`, `.AppImage`, `.app.tar.gz`)
- A `.sig` signature file
- A `latest.json` manifest file (generated once, contains all platforms)

All artifacts are uploaded to the GitHub Release.

## 8. Version Management

### Version Sync Requirement

**All three files must have the same version**:
1. `package.json` -- `"version": "1.2.0"`
2. `src-tauri/tauri.conf.json` -- `"version": "1.2.0"`
3. `src-tauri/Cargo.toml` -- `version = "1.2.0"`

Version mismatches will cause update detection to fail.

### Release Script (Automate Version Bumping)

```bash
#!/bin/bash
# scripts/release.sh
set -euo pipefail

BUMP_TYPE=${1:-patch}  # patch, minor, major

# Get current version from package.json
CURRENT=$(node -p "require('./package.json').version")

# Calculate new version
IFS='.' read -r major minor patch <<< "$CURRENT"
case "$BUMP_TYPE" in
  major) NEW_VERSION="$((major + 1)).0.0" ;;
  minor) NEW_VERSION="${major}.$((minor + 1)).0" ;;
  patch) NEW_VERSION="${major}.${minor}.$((patch + 1))" ;;
  *) echo "Usage: $0 [major|minor|patch]"; exit 1 ;;
esac

echo "Bumping $CURRENT -> $NEW_VERSION"

# Update package.json
npm version "$NEW_VERSION" --no-git-tag-version

# Update tauri.conf.json
cd src-tauri
sed -i.bak "s/\"version\": \"$CURRENT\"/\"version\": \"$NEW_VERSION\"/" tauri.conf.json
rm -f tauri.conf.json.bak

# Update Cargo.toml
sed -i.bak "s/^version = \"$CURRENT\"/version = \"$NEW_VERSION\"/" Cargo.toml
rm -f Cargo.toml.bak

# Regenerate Cargo.lock
cargo generate-lockfile --quiet
cd ..

# Commit and tag
git add -A
git commit -m "chore: bump version to v${NEW_VERSION}"
git tag -a "v${NEW_VERSION}" -m "Release v${NEW_VERSION}"

echo "Created tag v${NEW_VERSION}. Push with: git push origin main --tags"
```

### Update Channels (Stable / Beta / Nightly)

Use runtime endpoint configuration to support channels:

```rust
use tauri_plugin_updater::UpdaterExt;

async fn check_channel_update(app: &tauri::AppHandle, channel: &str) -> Result<()> {
    let url = format!(
        "https://releases.myapp.com/{channel}/{{{{target}}}}-{{{{arch}}}}/{{{{current_version}}}}"
    );

    let update = app
        .updater_builder()
        .endpoints(vec![url])?
        .build()?
        .check()
        .await?;

    if let Some(update) = update {
        // Handle update...
    }
    Ok(())
}
```

For GitHub-based channels, use separate repositories or release tags:
- Stable: `https://github.com/user/repo/releases/latest/download/latest.json`
- Beta: `https://github.com/user/repo-beta/releases/latest/download/latest.json`

Or use a custom server that routes based on the channel.

## 9. Rollback Strategies

### Tauri's Built-in Version Comparator

By default, Tauri only installs versions **newer** than the current one. To allow downgrades (rollbacks), override the version comparator:

```rust
let update = app
    .updater_builder()
    .version_comparator(|current, update| {
        // Install any version that differs from current (allows downgrades)
        update.version != current
    })
    .build()?
    .check()
    .await?;
```

### JavaScript `allowDowngrades` Parameter

The `check()` function in the JS API accepts an `allowDowngrades` option that achieves the same result from the frontend.

### Server-Driven Rollback

The update server controls what version to serve. To roll back:
1. Update `latest.json` to point to the older version
2. Users' apps will pick up the "downgrade" on next check
3. Requires `version_comparator` override or `allowDowngrades` enabled

### What Happens When an Update Fails

- **Download failure**: The app continues running the current version. No partial installs.
- **Signature verification failure**: Update is rejected entirely. App stays on current version.
- **Installation failure (Windows)**: The NSIS/MSI installer handles rollback internally. The previous version remains if install fails.
- **Installation failure (macOS/Linux)**: The `.tar.gz` is extracted to replace the app. If extraction fails, the existing binary may be corrupted. Consider keeping a backup.

### Defensive Strategies

1. **Always verify the update server is reachable** before starting download
2. **Use multiple endpoints** for redundancy
3. **Test updates in a beta channel** before pushing to stable
4. **Keep previous release artifacts** on GitHub/CDN for server-driven rollback
5. **Implement a "last known good version" config** in local storage

## 10. Common Pitfalls and Gotchas

### Version Mismatch
**Problem**: `package.json`, `tauri.conf.json`, and `Cargo.toml` have different versions.
**Fix**: Always use a release script that updates all three atomically.

### Missing Environment Variables
**Problem**: Build runs but produces no `.sig` files; updates fail signature verification.
**Fix**: Ensure `TAURI_SIGNING_PRIVATE_KEY` and `TAURI_SIGNING_PRIVATE_KEY_PASSWORD` are set in CI.

### Wrong URL in `latest.json`
**Problem**: The `url` field in `latest.json` points to the GitHub release page instead of the direct download link.
**Fix**: Use direct asset download URLs: `https://github.com/user/repo/releases/download/v1.2.0/MyApp.exe`

### Signature Field Contains Filename
**Problem**: The `signature` field in `latest.json` contains the `.sig` filename instead of its contents.
**Fix**: Read the `.sig` file contents and paste those into the `signature` field (the `tauri-action` handles this automatically).

### Windows Auto-Exit Surprise
**Problem**: On Windows, the app exits immediately when `install()` is called, without warning.
**Fix**: Use `on_before_exit` hook to save state, and warn the user before calling install.

### macOS Restart Not Automatic
**Problem**: On macOS/Linux, calling `app.restart()` or `relaunch()` does not happen automatically after install (unlike Windows).
**Fix**: Explicitly call `relaunch()` from the frontend or `app.restart()` from Rust after install completes.

### `createUpdaterArtifacts` Not Set
**Problem**: Build succeeds but no updater artifacts (`.sig`, `.tar.gz`) are generated.
**Fix**: Set `"createUpdaterArtifacts": true` in `tauri.conf.json` under `bundle`.

### GitHub Token Permissions
**Problem**: CI fails with "Resource not accessible by integration".
**Fix**: Go to repo Settings > Actions > General > Workflow permissions > "Read and write permissions".

### Lost Private Key
**Problem**: Private signing key lost or compromised.
**Impact**: Users with existing installs can NEVER receive updates again (the embedded public key won't match).
**Fix**: Back up private keys securely (e.g., 1Password, Vault). There is no recovery path.

### Notarization Adds Latency
**Problem**: macOS builds take 2-5 extra minutes for Apple notarization.
**Fix**: No workaround -- Apple needs time to scan binaries. Factor this into CI time budgets.

### Desktop-Only Plugin
**Problem**: Importing `@tauri-apps/plugin-updater` in a mobile build causes errors.
**Fix**: Gate the plugin registration with `#[cfg(desktop)]` in Rust and conditionally import in TypeScript.

### Stale `Cargo.lock`
**Problem**: After bumping version in `Cargo.toml`, the `Cargo.lock` is out of date.
**Fix**: Run `cargo generate-lockfile` after every version bump.

### Large Binary Sizes
**Problem**: Update downloads are large because Tauri bundles the full installer.
**Note**: Tauri does NOT support delta/differential updates. Every update downloads the full binary.
